# Documentation best practices

Checklist to run before publishing any Stratumn doc.

## Do

1. **Organize.** Clear headings/subheadings, bullets over walls of prose. Reader scans, doesn't read linearly.
2. **Make accessible.** Doc must live where the target reader actually looks — right Outline folder, not buried. No local-only copies.
3. **Keep current.** Update the doc when the thing it describes changes. A stale doc is worse than no doc — update in place rather than leaving old + new both live.
4. **Use visuals.** Diagram, screenshot, or table beats a paragraph when describing architecture, flows, or comparisons.
5. **Test on an outsider.** Before treating a doc as done, check: would someone unfamiliar with this system be able to follow it and succeed? If unsure, that's the doc failing.

## Avoid

1. **Jargon without context.** Don't assume the reader knows internal terms/acronyms. Define on first use or link out.
2. **Vagueness.** "Configure the service" is not a step. Give the exact command, file, flag, value.
3. **Skipping the test pass.** Don't publish unverified steps — a docs page with a broken command is worse than a missing page.
4. **Letting it rot.** Don't write once and forget. If you can't commit to updating it, say so in the doc (e.g. mark as snapshot/point-in-time).
5. **Hiding it.** Don't leave a doc as an unlinked draft or in the wrong collection where nobody will find it.
